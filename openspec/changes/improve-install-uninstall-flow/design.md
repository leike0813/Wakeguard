## Context

当前 `wakeguard install` / `wakeguard uninstall` 仅记录日志，不执行真实服务生命周期操作。  
同时首次安装缺少注册表健康检查与白名单初始化，导致状态不确定。  
此外，当前命令依赖手动路径执行，不满足全局 CLI 可用性要求。  
本次变更目标是形成可执行、可重复、可恢复且全局可调用的安装与卸载流程。

## Goals / Non-Goals

**Goals:**

- 通过命令完成 `Wakeguard` 服务安装与启动。
- 安装命令默认使用当前执行的 `wakeguard.exe`，并支持显式路径覆盖。
- 安装时将二进制复制到 `C:\Program Files\Wakeguard\bin\wakeguard.exe`。
- 安装时写入系统级 PATH：`C:\Program Files\Wakeguard\bin`（非 Windows 系统目录）。
- 安装后自动拉起 `wakeguard ui` 做白名单引导。
- 首次安装默认空白名单并提示用户选择设备加入或一键全选。
- 首次安装时基于字段白名单清理非法注册表项。
- 卸载时仅卸载 `Wakeguard` 服务并保留注册表数据。
- 卸载时移除 PATH 项并清理复制的二进制文件（必要时延迟删除）。

**Non-Goals:**

- 不构建 MSI/图形安装器。
- 不执行注册表整体删除。
- 不引入多服务卸载流程。
- 不在 Windows 系统目录（如 `System32`）放置可执行文件。

## Decisions

### Decision 1: 安装流程采用状态机编排

安装顺序固定为：
1. 解析安装源二进制路径（优先使用显式参数，否则使用当前执行文件）
2. 复制到 `C:\Program Files\Wakeguard\bin\wakeguard.exe`
3. 写入系统 PATH（若不存在）
4. 读取并校验注册表结构
5. 首次安装时清理非法注册表项
6. 初始化默认白名单（空）
7. 安装并启动 `Wakeguard` 服务
8. 拉起 `wakeguard ui`

选择理由：便于在每个阶段报告失败并保证可诊断性。

### Decision 2: 注册表匹配采用“字段白名单”规则

仅允许预定义键和值（如 `Whitelist`、`KnownDevices`、设备快照等）。  
首次安装发现未知键值时执行清理，避免历史脏数据影响行为。

选择理由：规则清晰、实现简单、便于测试。

### Decision 3: 卸载仅处理主服务

卸载流程仅停止并删除 `Wakeguard` 服务，不删除注册表数据。  
这样可支持后续重装保留用户选择与设备记录。

### Decision 4: 全局命令可达性由 Program Files + 系统 PATH 提供

统一使用 `C:\Program Files\Wakeguard\bin` 作为全局安装目录，避免写入系统目录。  
服务安装状态下，用户可直接使用 `wakeguard` 命令；卸载后移除 PATH 项使无路径调用失效。

### Decision 5: 运行中二进制采用延迟删除策略

由于 Windows 运行中的可执行文件无法直接删除，卸载时采用延迟清理策略：
- 先卸载服务并移除 PATH
- 再由短生命周期清理进程在主进程退出后删除 `wakeguard.exe`

## Risks / Trade-offs

- [风险] 安装后自动拉起 UI 在某些会话环境可能失败  
  → [缓解] UI 拉起失败不回滚服务安装，记录告警并提示手动执行 `wakeguard ui`。

- [风险] 白名单字段规则过严导致误删扩展数据  
  → [缓解] 规则集中定义并配套测试，新增字段需同步更新白名单。

- [风险] 服务已存在时重复安装  
  → [缓解] 安装前检测服务存在性，存在则走幂等更新路径或明确报错。

- [风险] PATH 变更后当前 Shell 不立即生效  
  → [缓解] 安装结果提示用户重开 Shell，且卸载后同样提示刷新会话。

- [风险] 卸载时二进制删除失败  
  → [缓解] 使用延迟删除并记录告警，必要时提供下次启动自清理。
