## Context

当前系统以后台服务为主，已具备设备扫描、禁用和策略能力，但缺少用户态交互入口。  
本次变更需要在不破坏服务主流程的前提下，引入最基础的主界面与弹窗交互，让用户可以直接管理白名单并处理首次出现的新设备。

## Goals / Non-Goals

**Goals:**

- 提供 `wakeguard ui` 命令，拉起主界面。
- 在主界面展示受管设备、白名单状态、唤醒禁用状态。
- 支持勾选方式增删白名单并立即持久化。
- 对首次发现且未记录的新设备弹窗确认是否加入白名单。
- 对已记录设备做到“再次插入不重复弹窗”。

**Non-Goals:**

- 不实现复杂多窗口和高级视觉主题。
- 不实现跨平台 UI。
- 不引入数据库或远程服务。

## Decisions

### Decision 1: 单二进制多模式

采用单可执行程序多子命令：`run`（服务）、`ui`（主界面）。  
这样部署简单，避免多进程版本漂移；同时满足“服务运行时可通过命令调出界面”。

备选方案：拆分成服务与 UI 两个独立二进制。  
未采用原因：部署与版本管理复杂度更高。

### Decision 2: 共享状态落注册表

白名单、已记录设备、设备快照、待处理弹窗事件均使用注册表持久化。  
服务负责写入最新设备状态与新设备事件，UI 负责读取展示并写回用户决策。

备选方案：本地文件。  
未采用原因：与既有“注册表作为配置存储”路线不一致。

### Decision 3: “已记录设备”作为弹窗去重主键

新设备判定基于 `stable_id`；当 `stable_id` 首次被发现时记录到 `KnownDevices`。  
后续同设备拔插重连不会再次触发弹窗，只更新设备在线/状态信息。

备选方案：仅以内存去重。  
未采用原因：服务重启后会丢失去重历史。

### Decision 4: UI 读模型与服务写模型解耦

新增 UI 状态读模型，避免 UI 直接依赖服务内部结构。  
服务输出最小稳定字段集（设备名、stable_id、白名单状态、禁用状态），UI 只消费读模型。

## Risks / Trade-offs

- [风险] 服务与 UI 并发访问注册表可能产生短暂不一致  
  → [缓解] 使用“全量快照覆盖写入 + 读取容错回退”策略。

- [风险] 弹窗过多影响体验  
  → [缓解] 仅首次未记录设备弹窗，记录后不重复。

- [风险] `stable_id` 误判导致错误去重  
  → [缓解] 沿用现有优先级策略，并保留管理页人工修正能力（勾选白名单）。
